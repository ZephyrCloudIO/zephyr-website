## The Shift in One Line

`vinext` gives teams a Next.js-style API on top of Vite. Zephyr makes that model production-ready at the edge.

That combination separates framework ergonomics from hosting assumptions.

## vinext: Next.js API Surface, Vite Engine

Cloudflare open-sourced `vinext`, a Vite-powered reimplementation of the Next.js API surface, built for edge runtimes.

What it unlocks:

- Familiar Next-style ergonomics
- Faster local builds and iteration through Vite
- Better alignment with edge-native runtimes
- Lower dependence on a vertically integrated hosting path

## Why Coupling Became the Problem

For years, production Next.js often implied Vercel as default deployment target. The pairing delivered speed and convenience, but also created tight coupling:

- Framework changes tied to platform rollout cycles
- Runtime capabilities gated by infrastructure choices
- Architecture shaped by vendor direction
- Cost behavior tied to one provider

Teams increasingly want optionality, predictable upgrades, federation compatibility, and long-term architectural control.

`vinext` makes decoupling feasible. Zephyr makes it operable.

## vinext + Zephyr Cloud: Dev to Global Edge

We paired `vinext` with Zephyr's Vite plugin and deployed a live demo.

Workflow:

1. Install `vite-plugin-vinext-zephyr`.
2. Add `withZephyr()` to `vite.config.js`.
3. Build and deploy.

No CI/CD rewiring. No custom edge worker scaffolding. No environment-specific rewrites.

Local build to global edge deployment in under one second.

Live demo: [https://lnkd.in/gp38k57p](https://lnkd.in/gp38k57p)

This is measurable deployment velocity, not a synthetic benchmark.

A Vite-first architecture changes that dynamic.

`vinext` + Vite + Zephyr supports:

- Predictable bundling behavior
- Cleaner remote entry handling
